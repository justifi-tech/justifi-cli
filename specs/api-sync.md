## Overview

A GitHub Actions workflow that keeps the CLI's vendored knowledge of the JustiFi API in sync with the upstream OpenAPI spec. The workflow watches `github.com/justifi-tech/public-docs`, bundles the multi-yaml OpenAPI source into a single document, regenerates Go types from it via `oapi-codegen`, computes a drift report listing operations the CLI does not yet expose as commands, and opens a pull request when anything changed.

This spec defines the workflow contract — not the codegen tool's internals. Commands themselves remain hand-written (see [http-client.md](http-client.md) for the rationale); only **types** and the **drift report** are generated.

**Scope:**
- The sync workflow: triggers, inputs, outputs, identity used to open PRs
- The vendored OpenAPI artifact in this repo and how it is bundled
- The `oapi-codegen` invocation that produces Go types under `internal/justifiapi/`
- The drift report format and where it lives
- How humans turn drift-report items into new command specs

**Out of scope:**
- The HTTP client itself — see [http-client.md](http-client.md). The generated types are *consumed* by the client; they do not replace it.
- Cobra command code generation — commands are hand-written.
- Build/release workflows — see [foundation.md](foundation.md).

**Dependencies:**
- [foundation.md](foundation.md) — workflow file naming, repo layout, `make generate` target.
- [http-client.md](http-client.md) — defines how generated types are consumed.

**Design principles:**
- **Generated code is a leaf, not the trunk.** We generate types and a drift report. We do not generate Cobra commands, client method signatures, or anything users see directly. This keeps codegen ugliness contained and reversible.
- **PRs, not pushes.** Every sync is a PR for human review. No auto-merge.
- **Drift report is the value.** A junior engineer or AI agent reading the drift report should be able to write the spec for a new resource group without re-reading the entire OpenAPI doc.
- **Vendor the spec.** A committed copy of the bundled OpenAPI lives in this repo. Builds are reproducible and offline.

## Specification

### Vendored OpenAPI artifact

A bundled OpenAPI document lives at `api/openapi-spec/justifi.json` in this repo. It is the single-file form of `github.com/justifi-tech/public-docs:openapi/multi-yaml/index.yaml`.

- The file is JSON (not YAML) for parser stability and diff legibility.
- The file is committed and reviewed like any other source. CI fails if `api/openapi-spec/justifi.json` is stale relative to the upstream commit recorded in its sibling `api/openapi-spec/source.json` (see below).
- A second file `api/openapi-spec/source.json` records provenance:
  ```json
  {
    "source_repo": "justifi-tech/public-docs",
    "source_ref":  "main",
    "source_sha":  "<commit sha of public-docs that produced the bundle>",
    "bundled_at":  "2026-05-26T12:34:56Z",
    "bundler":     "redocly@<version>"
  }
  ```

### Generated types

Generated Go types live under `internal/justifiapi/` and are produced from `api/openapi-spec/justifi.json` by `oapi-codegen`.

- Package path: `internal/justifiapi`.
- Generator config: `internal/justifiapi/oapi-codegen.yaml`. Output limited to `types` mode — request/response schemas only. No HTTP client is generated; that is hand-written (see [http-client.md](http-client.md)).
- All files under `internal/justifiapi/` are clearly marked generated (header `// Code generated by oapi-codegen. DO NOT EDIT.`) and are excluded from `gocyclo` and `revive` in `.golangci.yml`.
- `make generate` runs (in order): the bundler (to update `justifi.json`), then `oapi-codegen` (to update `internal/justifiapi/`). On a clean working tree, `make generate && git diff --exit-code` succeeds; CI uses this as its drift gate.

### Drift report

After regenerating types, the workflow compares the new OpenAPI operations against the set of commands actually registered in this repo, and writes `api/openapi-spec/drift.md`.

**How "registered commands" are discovered:** the codegen step also extracts the list of `(method, path)` pairs that the CLI claims to cover. The list is sourced from a hand-maintained `api/openapi-spec/coverage.yaml`:

```yaml
# Each entry maps an OpenAPI operationId to the CLI command that exposes it.
# Operations not listed here appear in drift.md as "uncovered".
operations:
  listPayments:    "payments list"
  getPayment:      "payments get"
  createPayment:   "payments create"
  capturePayment:  "payments capture"
  voidPayment:     "payments void"
  # ...
```

The drift report has three sections:

```markdown
# API Drift Report
_Generated 2026-05-26 against public-docs@<sha>._

## New operations (in upstream, not in CLI coverage)
| operationId | method | path | tag |
|-------------|--------|------|-----|
| listFooBars | GET    | /foo_bars | FooBars |

## Removed operations (in CLI coverage, not in upstream)
| operationId | last seen |
|-------------|-----------|
| ...         | ...       |

## Schema changes (operationId already covered, but request/response shape changed)
| operationId | change |
|-------------|--------|
| ...         | ...    |
```

Empty sections are still emitted with the heading and "_none_" placeholder, so the file shape is stable. The PR description includes the entire `drift.md` body inline.

### Workflow: `sync-openapi.yml`

Path: `.github/workflows/sync-openapi.yml`.

**Triggers:**
- `schedule:` daily at `06:00 UTC` (cheap; catches drift even when nobody fires manually).
- `workflow_dispatch:` with optional inputs:
  - `source_ref` (default `main`) — branch/tag/sha of `justifi-tech/public-docs` to bundle from.
  - `force` (boolean, default `false`) — open a PR even if no file changes (used for testing the workflow itself).

**Job steps:**

1. Check out `justifi-cli` at the default branch.
2. Check out `justifi-tech/public-docs@${source_ref}` into a sibling directory.
3. Run the bundler (`npx @redocly/cli@<pinned> bundle openapi/multi-yaml/index.yaml -o api/openapi-spec/justifi.json --ext json`).
4. Update `api/openapi-spec/source.json` with the upstream commit sha, timestamp, and bundler version.
5. If `git diff --quiet api/openapi-spec/ internal/justifiapi/` and `force == false`, exit cleanly (no-op).
6. Run `make generate` to regenerate types and the drift report.
7. Run `make lint && make test` to verify the working tree still builds and passes unit tests.
8. Open a pull request via `peter-evans/create-pull-request@v8` with:
   - Branch name: `sync-openapi/<source_sha-short>`.
   - Title: `chore(api): sync OpenAPI to public-docs@<source_sha-short>`.
   - Body: the full content of `drift.md`, followed by a "Generated by sync-openapi.yml" footer.
   - Labels: `api-sync`, `automated`.
   - Author: a dedicated GitHub App identity (`justifi-openapi-bot`). PAT fallback is acceptable for v1 if creating the App is not feasible yet.

**Failure handling:**
- If the bundler errors (malformed upstream), the workflow fails and posts an issue to the repo titled `sync-openapi failed: bundler` with the upstream sha. Do not open a partial PR.
- If `make generate` errors after a successful bundle, the workflow fails and posts an issue titled `sync-openapi failed: generate` — this indicates an `oapi-codegen` regression or a backwards-incompatible schema change we cannot machine-translate. Humans handle it.
- If `make test` fails, open the PR anyway (CI on the PR will display the failure for reviewers). Reasoning: a failed test on the PR is far easier to inspect than a swallowed workflow failure.

### CI drift gate

`test.yml` includes a job `openapi-fresh` that runs `make generate` and then `git diff --exit-code`. If the working tree has uncommitted changes after generation, the job fails with a message: `OpenAPI artifacts are stale; run \`make generate\` and commit the diff.` This prevents humans from editing generated files by hand or forgetting to regenerate after upstream changes.

### Coverage file maintenance

`api/openapi-spec/coverage.yaml` is hand-edited by humans when adding a new command. The rule is: **when a new resource spec moves from Draft to Ready, every operation it claims to expose must be added to `coverage.yaml`**. The drift report will then reflect the new coverage on the next sync.

A linter helper (`scripts/check-coverage.sh`) is invoked by `make lint` to verify:
- Every `operations:` key is a real operationId in `api/openapi-spec/justifi.json`.
- Every command listed (`"payments list"`) is reachable from `rootCmd` (resolved by running `justifi --help`-equivalent introspection).

## Notes

- **Why bundle instead of consuming multi-yaml directly.** `oapi-codegen` handles `$ref` across files inconsistently. Bundling first to a canonical single-file form (JSON for stable diffs) eliminates a class of "works on my machine" generator bugs and gives us a single artifact to commit and audit.
- **Why coverage.yaml is hand-maintained.** We could try to derive coverage from cobra command tree introspection alone, but the mapping from `(method, path) → "cobra path"` is a human-curated UX decision. Codifying it explicitly makes the drift report unambiguous.
- **Why daily, not on push.** Public-docs doesn't currently fire webhooks at us, and we don't want to depend on JustiFi infra changes. A daily cron is fine; we add `repository_dispatch` later if there's appetite to wire a webhook from `public-docs` to here.
- **Why JSON not YAML for the bundled spec.** JSON has deterministic key ordering when emitted by a stable serializer, and `git diff` on JSON is more legible than YAML diffs (no anchors, no folded strings).
- **Why no auto-merge.** Schema changes can be subtle (e.g., a field becoming nullable). A human PR review is the cheapest defense. The PR description carries the full drift report so the reviewer doesn't need to dig.
